## 1. きっかけ：Difyが「十分」だと勘違いしている現場が多すぎる

**Difyの売り：** ノーコード/ローコードでRAGチャットボットやワークフローを組める。

**現実：**

- 単一ターンのQ&Aか、決められたノード間の遷移しかできない
- 外部ツール（ブラウザ、ターミナル、ファイルシステム、VRChat等）への**直接的・非同期的実行が不可能**
- 長期記憶・パーソナライズは「ベクトルストアに突っ込む」程度で、忘却曲線やコンテキスト圧縮はサポート外
- マルチエージェント・委任（サブエージェント生成・オーケストレーション）は不可

**つまり：** Difyは「チャットボット作成ツール」に留まり、**「自律的に業務を完遂するAIエージェント」としては機能不足だ。**

---

## 2. Hermes Agent が目指すもの――「狭い腰、エッジで能力拡張」

Hermes Agent（Nous Research製）は、次の設計思想で構築されている。

| 原則 | 内容 | なぜ重要か |
|---|---|---|
| Per‑conversation prompt caching は神聖 | 同一会話内でプロンプトキャッシュを壊さない | トークンコストを劇的に削減、応答速度向上 |
| コアは狭い腰；能力はエッジに置く | ツールやプラグイン、スキルで機能拡張。コアに変更を加えても毎回のAPIコールにコストがかからない | 新機能を追加してもコスト増がほぼゼロ。スケールしやすい |
| マルチプラットフォームゲートウェイ | CLI・TUI・Desktop・Telegram・Discord・Slack・LINE・VRChat など同一コアで動作 | ユーザーがどこにいても同じエージェント体験 |
| 長期記憶・スキルシステム | Ebbinghaus 忘却曲線に基づくメモリ、Obsidian 連携、スキルマーケットプレイス | 過去の文脈を活かし、ドメイン知識を「プラグイン」として蓄積・共有 |
| サブエージェント委任（`delegate_task`） | 1タスク／バッチタスクをサブプロセスで実行し、結果をメイン会話に戻す | 複雑なリサーチやコードレビューなどを並列化し、コンテキスト肥大化を防止 |
| ツール実行の標準化 | `terminal`, `web_search`, `browser_navigate`, `vision_analyze`, `vrchat_*` など、ネイティブツールとして提供 | エージェントが「ファイルを開く」「ウェブを検索」「VR空間でアバターを動かす」など、実際の作業を自動化できる |

これらの機能は、Dify では**プラグインや外部サービス連携**としてしか実現できず、しかも「遅延」「状態共有」「セッション管理」の面で制約が多い。

## 3. 具体的な比較：Dify vs. Hermes Agent

| 項目 | Dify | Hermes Agent |
|---|---|---|
| 基本インターフェース | チャットUI（Web） | CLI / TUI / Desktop / ゲートウェイ（Telegram/Discord/…） |
| LLM呼び出しコスト | 1ターンごとにフルプロンプト送信（キャッシュなし） | 会話内プロンプトキャッシュにより、2回目以降はほぼトークン消費なし |
| 外部ツール連携 | プラグイン・Webhook で間接連携（同期的、制限あり） | ツールはネイティブ。`terminal(background=True, notify_on_complete=True)` 等で**非同期・バックグラウンド実行**が標準 |
| 長期記憶 | ベクトル検索（外部DB）に依存 | Ebbinghaus メモリ＋Obsidian連携。忘却曲線に基づく自動保持・忘却・復習 |
| マルチエージェント | 不可（単一ワークフロー） | `delegate_task` によるサブエージェント生成、`orchestrator` モード、ネスト可能（深度2） |
| マルチモーダル | 画像入力は一部プラグインに依存 | `vision_analyze`, `video_analyze`, `vrchat_*` など、標準ツールとして画像・動画・VR空間操作をサポート |
| 拡張性 | プラグインマーケットは若干、主に連携先サービス | スキルシステム（`skill manage`）で独自ロジック・ツール・プロンプトテンプレートをパッケージ化。公開・共有可能 |
| ガバナンス・監査 | ログは基本的 | セッションログ・ツール実行ログ・メモリ変更履歴が標準で取得可能。監査・コンプライアンスに強い |
| コスト感覚 | 使い放題プランでもトークン数に応じて課金が発生しやすい | キャッシュとエージェント設計により、同等の作業量でトークン消費を大幅削減 |

**結論：** Dify は「プロトタイプ作成」に適しているが、**本番運用・業務自動化・マルチエージェント協調**が必要になるとすぐに限界にぶつかる。その点で Hermes Agent は「開発段階から本番までシームレスにスケールできる」プラットフォームだ。

## 4. なぜ今、Hermes Agent を選ぶべきか――実務的なユースケース

### 4.1 データ収集・分析パイプライン（完全自動化）

```bash
# 1. ウェブから最新の業界レポートを取得
web_search(query="2026年 半導体市場 動向", limit=5) → results.json

# 2. 取得したURLをブラウザで開き、本文を抽出
browser_navigate(url=results[0].url)
browser_snapshot(full=true)
vision_analyze(...)

# 3. 抽出テキストを要約・構造化（LLM が処理）

# 4. 結果をNotionやSlackに自動投稿
terminal(command="notion-cli pages create ...", background=true, notify_on_complete=true)
```

この一連の流れを**単一のエージェントセッション**で完結させられる。Dify では各ステップごとに別のワークフローを作り、状態を受け渡すための仲介テーブルや Webhook が必要になる。

### 4.2 VRChat での自律アバター運営（はくあスタイル）

- `vrchat_autonomy_loop_tick` で観測・判断・行動をループ
- `vrchat_chatbox` でリアルタイムチャット、`vrchat_avatar_param` でジェスチャー制御
- バックグラウンドでは `web_search` で話題のトレンドを取得し、会話に自然に組み込む

これらはすべて**同じエージェントインスタンス**が担当。Dify＋外部サービスでは実現困難。

### 4.3 ナレッジベースの自律的育成（Obsidian連携）

- 会話ごとに重要事実を `memory` ツールで Ebbinghaus メモリに保存
- 夜間に cronjob で `memory_llm_wiki_export` を走らせ、Obsidian Vault に Markdown を自動出力
- 次日の会話では `memory` から関連情報を再呼び出し、文脈を「思い出す」

こうした**「記憶の生物学的サイクル」を模したナレッジマネジメント**は、Dify には存在しない概念だ。

## 5. 「Difyは死んだ」の真意――対象読者へのメッセージ

**誰に向けた主張か：**

- 「AIチャットボットを作っているけど、結局すぐ行き詰まる」エンジニア
- 「RAGだけでは顧客の本当のニーズ（自動レポート生成、ワークフロー自動化）が満たせない」コンサルタント
- 「社内にAIを根付かせたいが、ツールがバラバラで運用コストが高い」IT管理者

**伝えたいこと：**

- Dify は「プロトタイプフェーズ」のツールとしては良いが、**本番稼働を考えるならすぐに乗り換えるべき**
- 乗り換え先として Hermes Agent はオープンソースかつ拡張性が高く、コストパフォーマンスも圧倒的に優れている
- まずは `hermes tools` でツール一覧を確認し、`hermes agent` でちょっとしたタスクを動かしてみてほしい

---

**最後に：**

> 「AIエージェント」という言葉は、もはやチャットボットの延長線ではない。
> ツールを使い、状態を記憶し、仲間（サブエージェント）を作り、実際の仕事をこなす――
> それが真のエージェントだ。
> Dify から一歩踏み出すとき、そこに Hermes Agent が待っている。
