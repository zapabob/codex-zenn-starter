# 人間は何をレビューすべきか？ AI時代のコードレビューをSOPで設計し直す

AIエージェントがコードを書けるようになってから、開発組織の詰まりどころは少し変わりました。

これまでは、実装そのものが時間のかかる工程でした。仕様を読み、既存コードを調べ、実装し、テストを書き、動作確認する。そこに数時間から数日がかかるのは自然でした。

しかし、Codexのようなコーディングエージェントを導入すると、実装の速度は大きく上がります。小さな修正, テスト追加, リファクタリング, ドキュメント整備であれば、人間が手を動かすよりもずっと短い時間で差分が出てきます。

そのとき、次に詰まるのはどこでしょうか。

私は、かなり高い確率で **人間によるコードレビュー** だと思っています。

AI時代の開発では、コードを書く側よりも、コードを見る側が先に限界を迎えます。しかもそれは、レビュアーの能力が低いからではありません。人間の集中力、文脈保持、注意配分には限界があるからです。

この記事では、人間によるコードレビューがなぜボトルネックになるのかを整理し、AIとCIに何を任せ、人間が何を見るべきかを考えます。

## AI時代、ボトルネックは書く側から見る側へ移る

従来の開発では、1つのPRを作るまでに時間がかかりました。だから、人間のレビュー待ちが多少あっても、開発全体の中では許容されやすかったのです。

けれども、AIエージェントが実装速度を上げると、状況が変わります。

開発者がAIに依頼する。AIが数分で差分を作る。テストも追加する。ローカルで検証する。PRができる。

ここまでは速い。

ところが、その後に人間のレビュー待ちが発生します。レビュアーは別の作業中です。通知に気づいても、すぐには文脈を切り替えられません。ようやくレビューに入っても、差分が大きいと理解に時間がかかります。指摘が返ってきたころには、実装者も別のタスクに移っています。

この流れでは、AIが実装を速くしても、開発サイクル全体は速くなりません。

問題は、コード生成の速度だけを上げても、レビュー工程の設計が古いままだと、組織全体の速度は上がらないことです。

## 人間によるコードレビューが苦しくなる5つの要因

### 1. 非同期のタイムラグとコンテキストスイッチ

PRを出してから、実際にレビューが始まるまでには時間差があります。

この時間差そのものも問題ですが、より深刻なのは文脈の断絶です。実装者はPRを作ったあと、別の作業に移ります。数時間後、あるいは翌日にレビューコメントが返ってきます。その時点で、実装者は当時の判断、調査内容、迷った設計案をすぐには思い出せません。

つまりレビューの遅延は、単に待ち時間を増やすだけではありません。実装者とレビュアーの双方に、文脈を再構築する負荷を発生させます。

AI開発では、この問題がさらに目立ちます。

AIは短時間で差分を作れます。けれど、その差分を人間が数時間後に確認するなら、開発の律速は完全にレビュー側へ移ります。AIによって短縮された実装時間が、人間のレビュー待ちで失われます。

この対策は、人間が見る前に機械で確認できるものをすべて終わらせることです。

たとえば、`AGENTS.md` の `Done` セクションに次のような完了条件を書いておきます。

```md
## Done

変更後は必ず以下を実行すること。

- npm run lint
- npm test
- npm run build

失敗した場合は、失敗内容を要約し、修正してから再実行すること。
実行できない場合は、その理由と代替確認をPR説明に残すこと。
```

これだけで、人間のレビューは少し変わります。

レビュアーは、動くかどうか、フォーマットが崩れていないか、基本的なテストが通るかを最初から確認しなくてよくなります。見るべき対象を、仕様、設計、影響範囲に絞れます。

### 2. レビュアーの状態による指摘のムラ

人間のレビューにはムラがあります。

時間がある日は丁寧に見ます。忙しい日は浅くなります。疲れている日は細かい点に反応しやすくなります。あるPRでは通った書き方が、別のPRでは指摘されることもあります。

これは人間が悪いという話ではありません。人間は状態に左右されます。認知負荷が高い状況で、常に同じ粒度、同じ観点、同じ厳密さを保つのは難しいです。

AIエージェントが作る差分が増えると、レビュー対象の量も増えます。レビュー量が増えれば、人間の判断のムラも増えます。そうなると、レビューは品質保証 of the 場ではなく、個人の好みや疲労が混ざる場になってしまいます。

ここで必要なのは、レビュー観点の明文化です。

命名、責務分割、例外処理、ログ、テスト、ドキュメント更新。これらを毎回その場で考えるのではなく、`DESIGN.md` や `SOP.md` に落とします。

```md
## Review Policy

人間のレビュアーは、以下を優先して確認する。

- 要件に対して過不足がないか
- ドメインモデルが崩れていないか
- 既存の設計方針と矛盾していないか
- 将来の変更に対して不必要な制約を増やしていないか

以下は原則としてCIまたはAIレビューに委ねる。

- フォーマット
- import順
- 命名規則の軽微な違反
- lintで検出できる問題
- 単純な未使用変数
```

レビュー観点を明文化すると、人間の気分でレビュー品質が揺れにくくなります。さらに、AIにも同じルールを読ませられます。

人間とAIが同じ規律に従うこと。ここが大事です。

### 3. 些細な指摘に終始してしまう

コードレビューでよく起きるのが、表面的な指摘に時間を使いすぎることです。

変数名の好み、インデント、少し冗長な書き方、コメントの表現。もちろん、これらも一定の品質には関係します。ただし、レビューの中心に置くべきではありません。

AIが大きな差分を作る時代に、人間が表面的な指摘ばかりしていると、もっと重要な問題を見落とします。

本当に見るべきなのは、次のような点です。

* その変更は要求を正しく満たしているか
* 境界条件は扱えているか
* データモデルに無理がないか
* 既存機能への副作用はないか
* セキュリティや権限の前提を壊していないか
* 障害時の挙動が説明できるか

フォーマットや静的解析で拾えるものは、CIに任せるべきです。AIレビューで拾える一般的な不具合も、先にAIへ渡したほうがよいです。

人間のレビューは高価です。

高価な工程で、安価に自動化できる指摘をしてはいけません。

### 4. エージェント製コードの巨大なdiff

AIエージェントは、指示を少し広く解釈することがあります。

バグ修正を頼んだだけなのに、周辺のリファクタリングも始める。小さなUI変更を頼んだだけなのに、共通コンポーネントを追加する。テスト追加のつもりが、テストヘルパー全体を作り替える。

その結果、diffが大きくなります。

巨大なdiffは、人間にとってかなり負担が大きいです。変更意図、影響範囲、既存仕様との整合性をすべて追う必要があるからです。

そして怖いのは、巨大なdiffに対して、人間は次のどちらかになりやすいことです。

丁寧に見る時間が取れず、なんとなく承認する。
または、見るのが重くなり、レビューを後回しにする。

どちらも危険です。

対策は、AIに最初から変更範囲の制約を与えることです。

```md
## Rules

- 1回の変更は、依頼された目的に必要な最小範囲に限定する。
- 依頼に含まれないリファクタリングを行わない。
- 既存の命名や構造を大きく変える場合は、事前に理由を説明する。
- 100行を超える差分になりそうな場合は、作業を分割する提案を先に出す。
- 仕様変更、リファクタリング、テスト追加を同一PRに混ぜない。
```

AIには遠慮がありません。だからこそ、人間側が作業単位を設計する必要があります。

AIに自由に書かせるのではなく、レビューしやすい単位で書かせる。これが、AI時代の開発運用ではかなり重要になります。

### 5. なぜそうしたかが残らない

コードレビューで最も見落とされがちなのが、判断の履歴です。

PRには最終的な差分が残ります。しかし、なぜその設計にしたのか、なぜ別案を捨てたのか、どの制約を考慮したのかは残りにくいです。

人間が実装している場合は、レビュアーが質問できます。

なぜこの実装にしたのですか。
なぜここで例外を握りつぶしていますか。
なぜこのテーブルにカラムを追加したのですか。

しかしAIエージェントが実装した場合、判断の過程はチャット履歴に埋もれます。セッションが切れれば、次に読む人間にも、次に使うAIにも伝わりません。

これを放置すると、将来のレビューが重くなります。

だから私は、実装ログを残す運用を強く推します。

```md
_docs/2026-06-20_auth-session-timeout_codex.md
```

中身は、長い議事録である必要はありません。

```md
# auth-session-timeout 実装ログ

## 目的

セッション期限切れ時に、ユーザーが再ログインできるようにする。

## 変更内容

- APIレスポンスの401を検出
- セッション期限切れ専用のエラー型を追加
- UI側で再ログイン導線を表示
- 関連するユニットテストを追加

## 採用した判断

- グローバルなfetch wrapperで401を処理する
- 各画面コンポーネントでは認証エラーを個別処理しない

## 捨てた案

- 各API呼び出し側で個別に401を処理する案
  - 理由: 表示挙動が画面ごとに分散し、仕様変更時の修正漏れが起きやすいため

## 検証

- npm test
- npm run lint
- npm run build
```

このログがあるだけで、レビューはかなり変わります。

レビュアーは、差分だけではなく判断の背景を読めます。将来の開発者も、なぜその形になったのかを追えます。次回のAIエージェントも、過去の判断を踏まえて作業できます。

AI時代のレビューでは、コードそのものだけでなく、判断の保存が重要です。

## 人間はコードレビューを引退すべきか

ここまで読むと、人間のレビューは不要に見えるかもしれません。

でも、私はそうは思いません。

人間が見るべきものは、まだあります。むしろ、AI時代には人間のレビュー対象を狭めることで、人間の判断価値は上がります。

AIやCIに任せるべきものは、次の領域です。

* フォーマット
* lint
* 型チェック
* ビルド
* テスト実行
* 単純なバグ検出
* 一般的なセキュリティ指摘
* PR説明の下書き
* 変更影響範囲の要約

一方で、人間に残すべきものは次の領域です。

* 要件そのものが正しいか
* ユーザー体験として自然か
* ドメインモデルが適切か
* チームの設計方針と合っているか
* 将来の変更余地を不必要に潰していないか
* 組織として受け入れられる責務分割か
* その変更を運用で支えられるか

コードレビューという名前のままだと、人間はコードの細部を見ようとします。

しかし、AI時代に人間が見るべきなのは、コードそのものだけではありません。変更の意味です。判断の妥当性です。将来の開発への影響です。

人間はコードを全部読む人ではなく、変更の責任を判断する人になるべきです。

## SOP駆動でレビューを設計する

AI時代の開発では、気合いでレビュー品質を保つのは難しいです。

必要なのは、SOPです。標準作業手順として、AIと人間が同じ手順に従うようにします。

たとえば、リポジトリに次のようなファイルを置きます。

```text
.
├── AGENTS.md
├── DESIGN.md
├── SOP.md
├── _docs/
│   └── 2026-06-20_auth-session-timeout_codex.md
└── .github/
    └── workflows/
        └── codex-review.yml
```

それぞれの役割は明確にします。

`AGENTS.md` には、AIエージェントが守る作業ルールを書きます。
`DESIGN.md` には、設計原則やドメインモデルの考え方を書きます。
`SOP.md` には、PR作成前、レビュー前、マージ前の手順を書きます。
`_docs/` には、実装判断のログを残します。
GitHub Actionsには、テスト、lint、AIレビューを組み込みます。

この構成にすると、人間のレビュー前に多くの確認が終わります。

人間は、すでにテストが通っている差分、AIレビューを通過した差分、判断ログが残っている差分を見られます。

これは単なる自動化ではありません。

レビュー工程を、属人的な作業から、再現可能な工程へ変えるということです。

## レビューを自動化するだけでは足りない

ここで注意したいことがあります。

AIレビューを導入すれば、それだけでレビュー問題が解決するわけではありません。

AIレビューも、指示が曖昧なら曖昧なレビューをします。何を重要視するのか、どこまで指摘するのか、どの観点を無視してよいのかが決まっていなければ、人間レビューのムラがAIレビューに移るだけです。

だから、先に必要なのはツールではなく規律です。

どのチェックはCIに任せるのか。
どの指摘はAIに任せるのか。
どの判断は人間が持つのか。
AIが実装した場合、どのログを残すのか。
どのサイズを超えたらPRを分割するのか。

これらを決めてから、Codex、GitHub Actions、lint、test、静的解析を組み合わせるべきです。

順番を間違えると、AIはレビュー負荷を下げるどころか、レビュー対象を増やします。

## まとめ

AI時代の開発では、コードを書く速度は上がります。

しかし、人間のレビュー能力は同じ速度では増えません。集中力にも、記憶にも、文脈保持にも限界があります。

だからこそ、レビュー工程を設計し直す必要があります。

機械で見られるものは、機械に任せる。
AIで要約できるものは、AIに任せる。
人間は、要件、設計、責務、ドメイン、運用判断に集中する。
そして、その判断をSOPと実装ログとしてリポジトリに残す。

これからのコードレビューは、すべてのコードを人間が読む工程ではありません。

変更を安全に受け入れるための、規律ある判断工程です。

AIがコードを書く時代に必要なのは、より多くの人間レビューではありません。

人間が本当に見るべきものを決めることです。
