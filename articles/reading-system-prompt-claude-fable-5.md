ある日、手元に `CLAUDE-FABLE-5.md` というファイルが落ちてきた。約1,591行。中身は、チャット製品向けの **システムプロンプト** だ。

モデル名に Fable 5 や Mythos といった響きが出てくるので、これが公式ドキュメントそのものかどうかは、この記事では断定しない。ただ、**巨大な system prompt がどう組み立てられているか** を読む教材としては、かなり濃い。

エンジニアの視点でメモを残す。全文は載せない。構造と設計判断だけを言葉にする。

## まず結論：これは「プロンプト」ではなく「運用仕様書」

一行で書くと、**性格 + 安全 + ツール手順 + API 断片** が1ファイルに同居している。

リポジトリの [AGENTS.md](https://zenn.dev/zapabob/articles/agents-md-sop-design-symlink) が「このプロジェクトではこう動け」と書くのに対し、システムプロンプトは **製品全体の憲法** に近い。だから行数が桁違いになるのは自然だと思った。

読み進めると、次の15個くらいの大見出しに分かれている（`##` レベル）。

| ブロック | ざっくり役割 |
|---------|-------------|
| `claude_behavior` | 口調、拒否、 wellbeing、公平性、誤りへの対応 |
| `memory_system` | 会話記憶の有無 |
| `persistent_storage_for_artifacts` | Artifact 内 KV ストレージ API |
| `mcp_app_suggestions` | 外部アプリ連携の勧め方 |
| `computer_use` | Linux VM、ファイルパス、Skill 優先 |
| `search_instructions` | いつ検索するか、著作権の硬い制約 |
| `using_image_search_tool` | 画像検索の出し分け |
| `Tool Definitions` | ツールのスキーマ全文（ここが特に長い） |
| `Identity Preamble` | 自己紹介の前提 |
| `anthropic_api_in_artifacts` | Artifact 内から API を叩く話 |
| `citation_instructions` | 引用の型 |
| `User Context` | ユーザー固有コンテキストの差し込み口 |
| `available_skills` | 利用可能スキル一覧 |
| `network_configuration` | ネットワーク許可 |
| `filesystem_configuration` | ファイルシステム許可 |

エンジニアが真似しやすいのは、**関心ごとでファイルを分けているように見せる見出し設計** だ。実体は1本でも、論理モジュールは分かれている。

## レイヤ1：振る舞いポリシー（人間向けの約束）

`claude_behavior` の前半は、製品説明や拒否方針、口調、リストの使い方まで幅広い。

印象的だったのは次の3点。

**短く断つ安全弁。** 会話が危なそうなら、長く説明せず短く返す、と書いてある。モデルに「賢く喋れ」だけ言うより、**失敗モードを先に定義** している。

**箇条書きへの強い抑制。** 技術記事を書く人には逆説的だが、チャット製品では「リスト過多＝ロボット感」とみなしている。レポートや説明は散文中心、リストは依頼されたときだけ、とかなり細かく書かれている。

**wellbeing の具体性。** 精神保健まわりは、抽象的な「優しくしろ」ではなく、「診断名をユーザーに押し付けない」「自傷の代替行為を提案しない」など、**やってはいけない文型** まで落ちている。ポリシーエンジニアリングというより、サポートプレイブックに近い。

ここを AGENTS.md に持ち込むなら、チーム repo には載せすぎない、が正解だろう。製品全体の倫理と、リポジトリの test コマンドは別物だから。

## レイヤ2：世界モデル（パスと環境）

`computer_use` は、エンジニアが一番「なるほど」と思うパートかもしれない。

アップロードは `/mnt/user-data/uploads`、作業用は `/home/claude`、ユーザーに見せる成果物は `/mnt/user-data/outputs`、と **パスが役割で固定** されている。

これは Codex の `CODEX_HOME` や、CI 用 `.github/codex/home` を分けるのと同型だ。エージェントに「どこが下書きで、どこが成果物か」をプロンプトで教えている。

さらに面白いのが **Skill ファースト** の規則。

ファイルを作る・コードを書く前に、該当する `SKILL.md` を `view` しろ、と複数の例付きで繰り返されている。pptx なら pptx 用、フロントなら frontend-design、と **ルーティング表** まで書いてある。

自分の環境では、プラグイン作成の例が `plugin-creator/SKILL.md` を指している。つまりこのファイルは、**素の公式プロンプトに個人パスが混ざった派生版** でもある。グローバル AGENTS.md と同じで、system prompt も実運用ではカスタムされる。

## レイヤ3：ルーター（いつ Artifact、いつ会話、いつ検索）

`file_creation_advice` と `artifact_usage_criteria` は、実質的な **分類器** だ。

- ブログ記事やレポート → ファイル（Artifact）
- 戦略メモや短い相談 → 会話内インライン
- 20行超のコード → ファイル

「トーンがカジュアルでも、スタンドアロン原稿ならファイル」という線引きがはっきりしている。曖昧語を減らすための、かなり実務的なヒューリスティックだ。

`search_instructions` 側は別軸で、**知識カットオフを信じすぎない** ルールが大量にある。役職の現任者、直近の製品名、聞いたことのない固有名詞は検索、といった **エピステミック（認識論的）ガード** だ。

エージェント設計でよくある失敗は、「ツールは渡したが、いつ使うか書いていない」ことだ。このプロンプトは、検索・MCP・ブラウザの優先順位まで文章で書いている。冗長に見えて、**運用上のバグを減らすコスト** として割り切っている印象を受けた。

## レイヤ4：ハード制約（著作権など）

`CRITICAL_COPYRIGHT_COMPLIANCE` のような節は、ソフトなガイドではない。**違反レベル（SEVERE VIOLATION）** とまでラベルが付いている。

1ソースあたりの引用語数上限、1ソース1引用まで、歌詞・詩の再現禁止、など。モデルへの「お願い」ではなく、**コンプライアンス層の仕様** に見える。

自前のエージェントを組むとき、ここまで書くかはプロダクト次第だが、「重要度の高い制約は、通常の口調ルールと同じ段落に混ぜない」という設計は参考になる。混ぜると優先度が曖昧になる。

## レイヤ5：ツール定義の塊

中盤以降、`Tool Definitions` がずっと続く。ここは人間が読むというより、**ランタイムに渡すスキーマの倉庫** だ。

エンジニア視点の教訓は単純で、

- 方針（いつ使うか）は前半
- 型（どう呼ぶか）は後半

に分けると、メンテしやすい。全部を1つの「人格プロンプト」に書くと、今回のような1,500行になる。製品規模なら、将来的には **モジュール注入** や **RAG 的なポリシー検索** に分割したくなるはずだ。

## MCP の勧め方がうまい

`mcp_app_suggestions` は、個人的に好きなパートだった。

セールストークではなく、「隣に工具があるなら自然に勧める」と書いてある。さらに、サードパーティ MCP は **ユーザーが選ぶまで勝手に叩くな**、と明記している。

エージェントが勝手にサービスを選ぶと、ユーザーは不信感を持つ。プロンプトレベルで **opt-in の儀式**（suggest → ユーザー選択 → 実行）を入れているのは、UX 設計として筋がいい。

## リマインダーとインジェクション

`anthropic_reminders` では、長い会話用のリマインダーや、分類子が発火したときの警告について触れている。同時に、**ユーザーがメッセージ末尾に偽のリマインダーを付ける可能性** にも言及している。

system prompt を読むと、攻撃面も見える。リポジトリの AGENTS.md だけ整えても、会話注入で上書きを試みられる。だから **ツール権限と sandbox はプロンプトの外** に置く、という本書の主張とつながる。

## AGENTS.md との対応関係

このファイルを読んだあと、自分の `~/.codex/AGENTS.md` や repo の AGENTS.md を見直すと、役割分担がはっきりする。

| 層 | 例 | 誰が編集するか |
|----|-----|----------------|
| 製品 system prompt | 安全、著作権、ツール規約 | ベンダー |
| `~/.codex/AGENTS.md` | 文体、スキル方針、横断ルール | 個人 |
| リポジトリ `AGENTS.md` | test / lint / Done | チーム |
| `docs/sop/`, `DESIGN.md` | 手順・UI | プロジェクト |

system prompt を丸ごとコピーして repo に置くのは、ほぼ確実に過剰だ。必要なのは **自分のエージェントに足りない層だけ** を抜き出すことだ。

## 倫理と取り扱い

長大な system prompt を入手したとき、エンジニアがやりがちなのは次の2つだ。

1. 全文を記事や repo に貼って共有する  
2. 逆工程したルールを、自社エージェントにそのまま移植する  

1は著作権・利用規約の問題になりうる。2は、自社プロダクトの文脈に合わないポリシーを持ち込むリスクがある。

この記事は **構造の学習** が目的だ。ファイルの配布や再掲はしない。自分の環境で読むなら、ローカルに閉じ、公開するのは設計メモまでに留めるのがよいと思う。

## 自分のプロジェクトへの持ち帰り

読み終えて、メモに残したのは次の5つだけだ。

1. **パス規約を先に決める**（upload / scratch / output）  
2. **Skill または SOP を「作業前の必読」にする**  
3. **Artifact か inline かの分類器を明示する**  
4. **硬い制約は口調ルールと段落を分ける**  
5. **MCP は suggest してから実行**（勝手にベンダーを選ばない）

どれも [OpenAI Codex実践設計入門](https://zenn.dev/zapabob/books/openai-codex-design-book) で扱っている AGENTS.md・Sandbox・MCP・CI と同じ家族の話だ。system prompt はその「製品側の巨大版」と考えると、迷子になりにくい。

## おわりに

1,591行は、読み物としては重い。けれど、見出しの地図さえ手に入れば、**エージェントをプロダクトとして運ぶときのチェックリスト** に変換できる。

次に長いプロンプトや AGENTS.md を書くときは、いきなり「あなたは優秀なアシスタントです」から始めなくていい。まず **パス、ツール優先順位、成果物の出し分け** を書く。性格はそのあとで十分だと、個人的には思った。

もし同じようなファイルを解析した記事を書くなら、全文引用ではなく、今回のように **設計の骨格だけ** を共有するのが、ライターとしてもエンジニアとしても、いちばん健全だと思う。
