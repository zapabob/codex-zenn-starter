# SandboxとApprovalを設計する

Codexを安全に使ううえで、いちばん軽視してはいけないのが権限です。

Codexは、ファイルを読み、変更し、コマンドを実行できます。  
つまり、便利であるほど、誤操作や情報漏えいの影響も大きくなります。

## 権限設計の基本

最初は、次の順番で考えます。

1. 読むだけで足りるか
2. 作業ディレクトリ内だけ変更できればよいか
3. ネットワークアクセスは本当に必要か
4. 承認なしで実行してよいコマンドか
5. 失敗時に戻せる状態か

## 許可してよい作業の例

- テストの実行
- lintの実行
- formatterの実行
- 小さなソース修正
- ドキュメントの整備

## 慎重に扱う作業の例

- 依存関係の更新
- migrationの実行
- ネットワークアクセス
- 認証情報が絡む操作
- release assetの生成
- CI/CD設定の変更

## 禁止または明示承認にする作業

- `.env`や秘密鍵の表示
- working tree外への書き込み
- 本番DBへの接続
- 資格情報をログへ出す処理
- 破壊的な削除
- sandboxやapprovalの無効化

## この章の中心メッセージ

approvalは、Codexを疑うための仕組みではありません。  
人間が責任を持てる単位に作業を分けるための仕組みです。
