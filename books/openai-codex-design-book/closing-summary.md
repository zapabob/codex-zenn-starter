# Codex時代の開発作法をまとめる

この章では、本書全体を振り返り、日常開発に持ち帰れる **設計の型** をまとめます。

Codex は「賢い補完」ではなく、リポジトリ内で読み・書き・実行する **開発エージェント** です。その前提で、章ごとの学びをつなぎます。

## 本書の設計スタック

本書が提案する層は次の5つです。

```text
┌─────────────────────────────────────┐
│  CI / GitHub Actions（第7章）        │  ← 検証・PRレビューの自動化
├─────────────────────────────────────┤
│  MCP / エージェント定義（第6章）      │  ← 外部ツール・サブエージェント
├─────────────────────────────────────┤
│  Sandbox / Approval（第4章）         │  ← 権限の境界
├─────────────────────────────────────┤
│  AGENTS.md（第3章）                  │  ← リポジトリの作業規約
├─────────────────────────────────────┤
│  Codex CLI 導入・認証（第2章）        │  ← 実行基盤
└─────────────────────────────────────┘
```

プロンプトは最上層に毎回書くものではありません。**下の層が厚いほど、Codex の作業は安定** します。

## 章ごとの要点

| 章 | 持ち帰る一言 |
|---|-------------|
| 1 | Codex は環境設計の問題 |
| 2 | ChatGPT 認証を基本に、CI は別経路を選ぶ |
| 3 | コマンド・禁止事項・完了条件を AGENTS.md に書く |
| 4 | 日常は workspace-write、危険操作は Approval |
| 5 | フォークは upstream-first で読む |
| 6 | MCP は秘密を repo に置かず、失敗時は切り離せるように |
| 7 | CI は権限分離 + 本リポジトリ実装を `examples/` で配布 |
| 8 | 拡張はコアに直書きせず、検証ログを残す |
| 9 | 五層をセットで設計する（本章） |

## 認証の選び方（再掲）

| 場面 | 推奨 |
|------|------|
| ローカル開発 | ChatGPT サインイン（`codex login`） |
| 公開リポの CI | `OPENAI_API_KEY` + `codex-action`（proxy・drop-sudo） |
| 個人・非公開の CI | `CODEX_AUTH_JSON` + `codex exec review`（本リポジトリ例） |

いずれも **secret をジョブ全体の `env:` に晒さない** こと。第2章・第7章の共通ルールです。

## チームで始める最小セット

1. リポジトリルートに `AGENTS.md`（`examples/` からコピー）
2. `.codex/config.toml` またはプロジェクト設定で `sandbox_mode = "workspace-write"`
3. PR 時に Codex レビュー（`.github/workflows/codex-pr-review.yml` をコピー）
4. 必要になったら MCP を **1サーバーずつ** 追加（第6章）

最初から MCP 10個・フル自動 merge は不要です。AGENTS.md と Sandbox が効いてから広げます。

## フォークを使う人へ

公式 Codex で足りるなら、フォークは読むだけで十分です（第5章）。

次のときだけフォーク拡張を検討します。

- プラグインやエージェント定義を **製品として** 配布したい
- upstream のリリースサイクルに乗りながら独自機能を維持したい
- Windows 等、特定環境の検証ログを release に残す文化がある

拡張手順は第8章、`examples/AGENTS.md.codex-fork.md` が出発点です。

## 本リポジトリで試す

| パス | 用途 |
|------|------|
| `examples/AGENTS.md.*.md` | スタック別 AGENTS.md テンプレ |
| `examples/.github/` | CI 用 config・workflow・プロンプト |
| `AGENTS.md` | 本書リポジトリ自身のルール（dogfooding） |
| `.github/workflows/codex-pr-review.yml` | ChatGPT 認証 PR レビュー（動作実績あり） |

`npm run preview` で全章をローカルプレビューできます。Zenn は `main` への push でデプロイされます。

## この本のゴール（再掲）

読者が次を **自分のリポジトリで再現** できる状態を目指しました。

- [ ] Codex に任せる範囲と人間がレビューする範囲を分けられる
- [ ] AGENTS.md で test / lint / 禁止事項を明文化できる
- [ ] Sandbox と Approval を用途別に選べる
- [ ] MCP を安全に足せる（または足さないと判断できる）
- [ ] CI に Codex を載せ、secret を守れる
- [ ] 公式とフォークの関係を説明できる
- [ ] フォーク拡張をコアに直書きせずに足せる

## おわりに

Codex を起動する本ではなく、**Codex が安全に働ける開発環境を作る本** —— 第1章の約束どおり、ここまでの章で一通りの部品は揃いました。

あとは自分のリポジトリに `AGENTS.md` を1枚置き、Sandbox を1段階だけ厳しく（または緩く）設定し、1本の CI ジョブを試すことから始めてください。うまく回ったら MCP やフォーク拡張を足す。それが本書の推奨する順序です。
